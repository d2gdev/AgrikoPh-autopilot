## Approval-ready issue list

### GSC-01 — Autopilot constructs overlapping 29-day "28-day" windows

Priority: P1

Google Search Analytics dates are inclusive. `fetch-seo-data` subtracts 28 days and then sends both endpoints, producing 29 calendar days. Its prior period ends on the current period's first date, so the two periods overlap by one day. `fetch-gsc-data` has the same 29-day construction.

This also explains the authenticated SEO Pilot message `no non-overlapping comparison period yet`: both normalized and raw comparison readers correctly require the prior period to end before the current period starts, but the fetch jobs persist an overlapping boundary.

Proposed remediation:

- calculate inclusive 28-day windows as `end - 27 days` through `end`;
- set the prior end to one day before the current start;
- use the same corrected helper in both GSC jobs;
- add regression coverage for calendar-day count, adjacency, and the displayed comparison; and
- refresh normalized/raw snapshots before verifying the authenticated UI.

### GSC-02 — SEO Pilot labels query/page evidence sums as property totals

Priority: P1

For the exact 2026-06-18–2026-07-16 window, the API property aggregate is 201 clicks and 32,488 impressions. The normalized query/page evidence sums to 51 clicks and 13,402 impressions, which is what the authenticated SEO Pilot displays as `Total Clicks` and `Impressions`.

The query/page dimension omits anonymized and otherwise ungrouped data. It is appropriate for opportunity evidence but not for property-total cards. The UI currently understates clicks by 74.6% and impressions by 58.7% for that exact window.

Proposed remediation:

- fetch and persist one dimensionless property aggregate for each reporting window;
- keep query, page, and query/page rows for evidence and ranking analysis;
- feed the four total cards from the aggregate, with explicit provenance; and
- verify the displayed values against a same-window direct API response.

### GSC-03 — Homepage structured data contains an incomplete OfferCatalog

Priority: P1

The latest URL Inspection response reports invalid Product snippets on the homepage: four product items lacked all of `offers`, `review`, and `aggregateRating`.

The live theme read-back and rendered page trace the cause to the homepage `OfferCatalog`. Its `Offer` entries have no price, currency, availability, or URL, and point to product-like items. The live theme asset is older than the local theme source, so remediation must start from a live-asset read-back and avoid overwriting unrelated local work.

Proposed remediation:

- remove the incomplete `OfferCatalog`/`hasOfferCatalog` relationship from the homepage;
- retain the valid navigational `ItemList`;
- validate the rendered JSON-LD and URL Inspection result; and
- push only the exact theme asset through the Admin API with checksum read-back and rollback bytes.

No rating or review markup may be invented.
