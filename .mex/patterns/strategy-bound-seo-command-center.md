# Strategy-bound SEO Command Center

Executable non-blog synchronization must create or link an exact pending Recommendation. Stable task identity excludes proposal bytes; keep the proposed hash as separate execution evidence. Confirmation only approves/queues, while `execute-approved` revalidates and serializes the normalized Shopify target before mutation.

Once the linked Recommendation is approved, override-approved, or executing, synchronization must not overwrite task bytes. Approval freezes the strict hash, and dispatch compares it before any lock, observation, or mutation.

## Context

SEO observations are evidence, not governance. Only the active `agrikoph.com` topical-map projection may classify a missing page or absent required link as promotable. The strategy version ID, package SHA-256, exact rule IDs/domains, and safe source references travel together from projection through analysis and proposal persistence.

## Steps

1. Load the command center through the authenticated, private/no-store route; never expose raw artifacts or compiled payloads.
2. Project only bounded rule policy fields (`resolutionStatus`, conditions, evidence requirements, review requirements). Treat missing policy, `manual_gate`, `activation_blocking`, and conditions without explicit satisfied evidence as non-actionable at analysis, selected reconstruction, and compliance revalidation.
3. Persist analysis in the versioned envelope with the exact active version ID and package hash.
4. On read, return `stale` with `analysis: null` when the cached identity differs; never leak or render the old findings.
5. Keep unmapped GSC demand in the separately labelled observation view. Do not provide a promotion action until an active map rule governs it.
6. For content or internal-link promotion, reconstruct the candidate from the current server projection and reject stale identity, unrelated rule IDs, altered evidence, or incomplete link pairs before persistence.
7. Persist a deterministic candidate ID from strategy identity, kind/action, normalized URLs, and sorted rule IDs. Selected promotion sends the exact analysis timestamp/strategy identity plus at most 100 IDs, reloads current server evidence, and commits each candidate independently through the existing evaluator/compliance/dedupe transaction.
8. Redirects use the existing Store Task lifecycle only for create-when-exact-source-is-absent. Matching redirects are satisfied; conflicting exact sources stay advisory and are never updated automatically. Canonicalization and indexation remain advisory-only.
9. When removing a legacy strategy surface, delete its module, unreachable panels/handlers/tabs, and add a recursive runtime-source regression scan so hidden fallback code cannot return.
10. After the exact active-map analysis snapshot is persisted, synchronize non-blog Store Tasks as an independent best-effort step. Return only bounded counts, preserve analysis readiness on sync failure, and keep the standalone sync route for operator retry.
11. Project the compiled page title separately from `primaryKeywordOrTheme`: the title drives the Content Gap/proposal heading, while the keyword remains the target query. Normalize P0/P1/P2/P3 only for UI bands and badge tones; preserve original priority strings in advisory Store Tasks.
12. Classify non-blog page mutations from explicit decision language only. Generic keep/preserve/indexation/publication/conditional/optimize/strengthen language creates no executable body or metadata task. Reuse that classifier during execution revalidation.
13. Refresh bounded priority/canonical/decision/evidence fields on an existing pending or failed canonical/index advisory without changing semantic identity; a second identical synchronization must be unchanged.
14. Fail closed on internal-link instructions that embed an unmet condition in `currentBodyState` or `requiredAction`, even when the compiled policy status is `resolved`. Append-link work also requires explicit `add` or `ensure` intent; retain/replace/remove and missing instructions remain neutral. Apply both checks in analysis, selected reconstruction, compliance evaluation, non-blog Store Task projection, operator presentation, and final execution rule revalidation.
15. Keep large grouped link tasks listable: list DTOs return at most 25 rule IDs plus the exact `ruleCount`; authenticated detail retains up to 100 complete rules and links.
16. Persist and display the map-recorded link state separately from the fresh Shopify observation. Canonical/indexation advisories retain publishing state alongside priority, decision, evidence, and proposed canonical URL.
17. Project phase and schedule authority through explicit bounded summaries. Schedule obligations remain proposal-only; elapsed time never authorizes action, and global gates must not appear under a generic blocked-state filter.
18. Keep gated content visible as non-selectable operator context, retaining bounded current-title and observation evidence when available. New-content drafts must return the exact persisted active-map title, and new-content/refresh prompts should consume the bounded persisted secondary variants rather than silently dropping them.
19. Derive operational proposal priority from the current active-map rule during selected reconstruction, not from the cached analysis row. Map P0/P1 to proposal P1, P2 to P2, and P3 to P3 while retaining the original map priority in review evidence.

## Gotchas

- A valid rule ID is insufficient if it belongs to a different projected candidate.
- Client-authored map evidence is untrusted even when strategy identity matches.
- Blog refresh candidates remain governed by exact active-map reconstruction. Non-blog Store Tasks require an explicit metadata or body-content instruction; generic optimize/strengthen wording is insufficient.
- A newer active pointer invalidates cached analysis immediately; timestamps alone do not establish freshness.
- Local builds must use the exact non-production `autopilot_test` URL and include `connection_limit` and `pool_timeout`; never source production credentials for a build.
- Inspection completeness must use the same executable scope as analysis. Non-blog Shopify objects that Content Pilot deliberately suppresses must remain visible as unsupported findings, but must not make fully inspected blog actions globally unavailable.
- Never route governed blog articles into Store Tasks; blog create/refresh/link work remains in Content Pilot, while Store Tasks cover supported non-blog Shopify resources.
- Treat a blog article as `(blogHandle, handle)` everywhere. Normalize exact `/blogs/<blogHandle>/<handle>` paths for absence, links, reconstruction, and publish revalidation; a same-handle article in another blog is not evidence.
- Resolve an exact governed blog source to `(blogHandle, handle)` before invoking draft AI. If the exact article is unavailable, fail the draft closed; never select a same-handle article from another blog or continue with source-less current content. Preserve handle-only lookup only for legacy proposals that have no exact blog URL.
- Shopify resource `updatedAt` detects source changes; local `capturedAt` establishes observation freshness. For AI drafts, keep each request at 25 candidates but iterate every deterministic chunk so later candidates are not starved.
- Do not label redirect rules candidate, blocked, satisfied, or conflicting from compilation alone. SEO Pilot points operators to Store Pilot; only its exact-source observation may assign the operational state.
- Do not turn a missing blog-link candidate into a blocker. Use persisted suppression evidence for unavailable observations, route non-blog sources to Store Pilot, and otherwise show a neutral no-candidate state.
- Never reconstruct an internal-link destination from a handle. Persist, prompt, validate, publish, and reconcile the exact normalized `fromUrl`/`toUrl`; blog handles are display context only.
- The body-authored `/api/seo/gaps/promote` map path is retired. Do not add a client fallback; selected promotion sends candidate IDs and exact analysis identity to `/api/seo/gaps/promote-selected`.
- Conditional wording in an internal-link payload is not condition evidence. Until the contract supplies explicitly satisfied evidence, show `Gate required` and persist no executable proposal or Store Task.
- Revalidate current rule policy and mutation intent after approval and before locking or observing Shopify. Matching rule IDs and targets alone do not keep an older approved page, link, or redirect task executable.
- Do not pass an unbounded grouped `ruleIds` array through the Store Task list DTO. Bound the preview while preserving the exact count and complete authenticated detail.
- Persisted redirect-conflict hashes are operator evidence, not implementation detail; project and display the exact bounded hash with the observed target, redirect ID, and capture time.
